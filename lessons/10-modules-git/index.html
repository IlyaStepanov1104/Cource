Модули

Разбиваем код на файлы

Проблема

Игра на 500 строк в одном main.py

  • найти нужное - сложно
  • переиспользовать класс в другом проекте - невозможно
  • работать вдвоём - конфликты на каждой строке
Решение: разбить на несколько .py файлов

Модуль = файл .py

Любой .py уже является модулем

project/
  main.py
  hero.py       # модуль "hero"
  enemy.py      # модуль "enemy"
  utils.py      # модуль "utils"

Имя модуля = имя файла без .py

import

# hero.py
class Hero:
    def __init__(self, name):
        self.name = name

def greet(hero):
    print(f"Привет, {hero.name}!")
# main.py
import hero

h = hero.Hero("Артур")
hero.greet(h)

Три формы импорта

import модуль

import hero
h = hero.Hero("A")

доступ через hero.X

from ... import ...

from hero import Hero
h = Hero("A")

берём только нужное

import ... as ...

import numpy as np
np.array([1,2,3])

короткое имя-алиас

Опасный вариант

from hero import *   # тащит ВСЁ
  • не видно, что именно пришло
  • имена могут перезаписать твои
Не используй import * в своих проектах

Пример: разбиваем ДЗ про RPG

rpg/
  main.py       # запуск боя
  character.py  # Character + Warrior/Mage/Archer
  team.py       # класс Team
# main.py
from character import Warrior, Mage, Archer
from team import Team

team = Team([Warrior("Арагорн"), Mage("Гэндальф")])
team.attack_enemy(Archer("Леголас"))

Нужна изоляция пакетов по проектам

Виртуальное окружение

# создать venv в папке проекта
python -m venv .venv

# активировать (Windows)
.venv\Scripts\activate

# активировать (Linux/Mac)
source .venv/bin/activate

# теперь pip install ставит только в этот venv
pip install requests

.venv - обычная папка внутри проекта, её добавляют в .gitignore

Правило

1
Новый проект -> сразу python -m venv .venv
2
Активировать venv
3
Ставить пакеты через pip install
4
Перед коммитом pip freeze > requirements.txt

Полный цикл: новый проект

mkdir game && cd game

python -m venv .venv
.venv\Scripts\activate          # Windows
# source .venv/bin/activate     # Linux/Mac

pip install pygame rich

# пишем код...

pip freeze > requirements.txt

Частая ошибка

$ pip install requests

Проверяй: в начале строки терминала должна быть (.venv)

Git

Машина времени для кода

Зачем VCS

  • откатить изменения, если сломал
  • видеть, что и когда менял
  • работать параллельно с другими
  • иметь копию в облаке
Без Git серьёзный проект не делают нигде

Git vs GitHub

Git — это инструмент (программа), который хранит историю кода

GitHub — это сайт, где лежат Git-репозитории

Git

  • работает у тебя на компьютере
  • хранит историю
  • коммиты, ветки, merge
  • можно без интернета

GitHub

  • хостинг для Git-репозиториев
  • хранит код в облаке
  • нужен для совместной работы
  • PR, Issues, CI
GitHub — это не Git, это сервис поверх Git

Git хранит снимки

Коммит - это снимок всего проекта в момент времени

Каждый коммит знает родителя -> получается цепочка

Три области Git

Первая настройка

Один раз на весь комп - сказать Git, кто ты

git config --global user.name "Ilya Stepanov"
git config --global user.email "ilya@example.com"

git config --global init.defaultBranch main

# проверить
git config --list

Имя и почта попадают в каждый твой коммит

Создаём репозиторий

cd my-project
git init
# Initialized empty Git repository in ./.git/

Появилась скрытая папка .git/ - там живёт вся история

Удалишь .git/ - удалишь всю историю проекта

status и add

git status
# Untracked files:
#   hero.py
#   main.py
git add hero.py main.py
# или всё разом:
git add .
git status
# Changes to be committed:
#   new file: hero.py
#   new file: main.py

Первый коммит

git commit -m "add Hero class"
# [main (root-commit) 7d32786] add Hero class
#  2 files changed, 25 insertions(+)

Ещё коммиты

# правим hero.py
git add hero.py
git commit -m "add greet method"

# правим main.py
git add main.py
git commit -m "print hero greeting"

Что именно поменялось: git diff

$ git diff
diff --git a/hero.py b/hero.py
- hp = 100
+ hp = 150

- удалили, + добавили

git diff            # рабочая папка vs staging
git diff --staged   # staging vs последний коммит
git diff HEAD~1     # последний коммит vs предыдущий

Откатить изменения

# испортил файл, ещё не делал add
git restore hero.py        # вернуть как в последнем коммите

# уже сделал add, но передумал
git restore --staged hero.py   # убрать из staging

# вообще хочу откатиться на 1 коммит назад
git reset --hard HEAD~1    # ОСТОРОЖНО: потеряешь изменения

Откатить опубликованный коммит

Коммит уже ушёл на GitHub, reset использовать нельзя

git revert 7d32786
# создаёт новый коммит, который отменяет указанный

Зачем ветки

Хочу пробовать новую фичу и не сломать рабочий код

-> создаю ветку, экспериментирую там

-> если получилось - вливаю в main

-> если не получилось - удаляю ветку

Команды для веток

git branch список веток
git branch feature создать ветку
git checkout feature переключиться на ветку
git checkout -b feature создать и переключиться
git branch -d feature удалить (если влили)
git branch -D feature удалить силой

Создаём ветку

git checkout -b feature-sword
# переключились на новую ветку feature-sword

Коммитим в ветку

# пишем код, коммитим
git commit -am "add Sword class"
git commit -am "hero wields sword"

Полная работа с веткой

git checkout main
git checkout -b feature-sword

# правим файлы...
git add .
git commit -m "add Sword class"

# правим ещё...
git commit -am "hero wields sword"

git checkout main
git merge feature-sword
git branch -d feature-sword       # больше не нужна
git push

Вливаем ветку: merge

git checkout main
git merge feature-sword

M - merge-коммит с двумя родителями

Rebase: переписать историю

git checkout feature-sword
git rebase main

Rebase берёт коммиты ветки и заново накладывает поверх main

Merge vs Rebase

Merge

История ветвится, виден факт слияния

Rebase

История линейная, хэши пересоздаются

Когда что

Merge

  • коммит уже видели другие
  • хочешь сохранить "историю как было"
  • main -> feature

Rebase

  • только локальная ветка
  • хочешь линейную историю
  • feature -> актуальный main
Золотое правило: не ребазь то, что уже запушил и увидели другие

Конфликты

Обе ветки поменяли одну и ту же строку

<<<<<<< HEAD
damage = 10
=======
damage = 15
>>>>>>> feature-sword
1
открыть файл, выбрать правильную версию
2
git add файл
3
git commit (или git rebase --continue)

Если всё плохо

git merge --abort     # отменить начатый merge
git rebase --abort    # отменить начатый rebase

# всё вернётся к состоянию до команды
Запутался в конфликтах - abort и попробуй снова

git stash: отложить изменения

Надо срочно переключиться на другую ветку, но код не готов коммитить

git stash              # спрятал всё незакоммиченное
git checkout main        # переключился
# ...сделал срочное дело, закоммитил...

git checkout feature
git stash pop          # вернул отложенные изменения

git stash list         # посмотреть все заначки

GitHub

Код в облаке

Remote: удалённая копия

Удалённый репозиторий (remote)

Это копия проекта в интернете

git remote add origin https://github.com/user/repo.git
origin — это просто имя удалённого репозитория

Связываем локальный репо с GitHub

# создать репо на github.com -> получить URL
git remote add origin https://github.com/user/my-project.git
git push -u origin main

Флаг -u запоминает связь: дальше просто git push

С нуля: проект на GitHub

mkdir my-game && cd my-game

# 1. Python
python -m venv .venv
.venv\Scripts\activate
echo ".venv/" > .gitignore

# 2. Git локально
git init
git add .
git commit -m "initial commit"

# 3. GitHub (после создания пустого репо на сайте)
git remote add origin https://github.com/user/my-game.git
git push -u origin main

push и pull

push

git push

отправить локальные коммиты на GitHub

pull

git pull

забрать новые коммиты с GitHub

clone: взять чужой проект

git clone https://github.com/user/some-repo.git
cd some-repo
# всё готово: код, история, remote уже настроен

Так ты скачиваешь любой open-source проект с GitHub

Pull Request

Запрос на слияние: "посмотрите мой код и влейте в main"

git checkout -b fix-bug
# правим, коммитим
git push -u origin fix-bug

# идём на github.com -> Compare & pull request
# кто-то ревьюит -> merge -> ветка вливается в main
Так работают все open-source проекты и команды

Типичный день в команде

git checkout main
git pull                         # забрать свежее

git checkout -b feature-login      # новая ветка
# пишем код...
git add .
git commit -m "add login form"

git push -u origin feature-login # залить на GitHub
# создать Pull Request на сайте
# ревью -> merge -> удалить ветку

Что запомнить

  • Модуль - это .py файл, который импортируют
  • pip install + venv + requirements.txt
  • Коммит - снимок проекта; цепочка = история
  • Ветки дают безопасный эксперимент
  • Merge сохраняет историю; rebase делает её линейной
  • GitHub - удалённая копия, push/pull/clone
