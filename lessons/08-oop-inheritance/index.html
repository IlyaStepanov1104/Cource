Идея: общий класс-родитель

Что общего у собаки, кошки и птицы? Они все - животные.
Общее вынесем в класс Animal, остальное - в наследники.
Наследование - способ создать новый класс на базе существующего

Синтаксис

class Animal:
    def __init__(self, name, age):
        self.name = name
        self.age = age

    def sleep(self):
        print(f"{self.name} спит")


class Dog(Animal):  # Dog наследуется от Animal
    pass
class Dog(Animal): - в скобках родительский класс

Что получил наследник

class Animal:
    def __init__(self, name, age):
        self.name = name
        self.age = age
    def sleep(self):
        print(f"{self.name} спит")

class Dog(Animal):
    pass

rex = Dog("Рекс", 3)   # __init__ от Animal
rex.sleep()            # sleep от Animal - "Рекс спит"
print(rex.name)        # "Рекс"
Наследник получает все атрибуты и методы родителя бесплатно

Наследник добавляет своё

class Animal:
    def __init__(self, name, age):
        self.name = name
        self.age = age

class Dog(Animal):
    def bark(self):

class Cat(Animal):

Dog("Рекс", 3).bark()   # Рекс: Гав!
Cat("Мурка", 2).purr()  # Мурка: Мрррр
Общее - в родителе, уникальное - в наследниках

isinstance и issubclass

class Animal: pass

Проблема: потеряли родителя

class Animal:
    def __init__(self, name, age):
        self.name = name
        self.age = age

class Dog(Animal):
    def __init__(self, name, age, breed):

Переопределили __init__ - и забыли про name и age

super() - ссылка на родителя

class Animal:
    def __init__(self, name, age):
        self.name = name
        self.age = age

class Dog(Animal):
    def __init__(self, name, age, breed):
        super().__init__(name, age)  # вызов __init__ родителя
        self.breed = breed

rex = Dog("Рекс", 3, "Хаски")
print(rex.name, rex.age, rex.breed)
# Рекс 3 Хаски
super().method() - вызов метода родителя без дублирования кода

super() в любом методе

class Animal:
    def describe(self):
        return f"Животное по имени {self.name}"

class Dog(Animal):
    def describe(self):
        parent = super().describe()   # родительское описание
        return f"{parent}, порода {self.breed}"

rex = Dog("Рекс", 3, "Хаски")
print(rex.describe())
# Животное по имени Рекс, порода Хаски
Берём результат родителя и дополняем его своим

Пример: транспорт

class Vehicle:
    def __init__(self, brand, max_speed):
        self.brand = brand
        self.max_speed = max_speed
    def info(self):

class Car(Vehicle):
    def __init__(self, brand, max_speed, seats):
        super().__init__(brand, max_speed)

    def info(self):

print(Car("Toyota", 200, 5).info())

MRO - порядок поиска метода

Если метод есть у нескольких родителей - Python ищет слева направо. 
class A:
    def hi(self): print("A")

class B:
    def hi(self): print("B")

class C(A, B):  # сначала A, потом B
    pass

C().hi()               # "A"
print(C.__mro__)       # порядок поиска

Полиморфизм

Разные объекты - один интерфейс

Полиморфизм

Poly (много) + morph (форма) = "много форм".
Один и тот же вызов работает с разными классами по-разному

Пример: ферма

class Dog:
    def speak(self): print("Гав!")

class Cat:
    def speak(self): print("Мяу")

class Cow:
    def speak(self): print("Муу")

farm = [Dog(), Cat(), Cow()]

for animal in farm:
    animal.speak()       # у каждого свой speak
Один код - разные реализации. Это и есть полиморфизм

Duck typing

"Если крякает как утка и ходит как утка - это утка"
Python не проверяет тип - проверяет наличие метода. 
def make_sound(thing):
    thing.speak()    # главное, чтобы был .speak()

make_sound(Dog())     # работает
make_sound(Cat())     # работает
make_sound(Robot())   # тоже работает, если есть speak

Преимущество: легко расширять

class Duck:
    def speak(self): print("Кря!")

farm = [Dog(), Cat(), Cow(), Duck()]  # добавили утку

for animal in farm:
    animal.speak()    # код цикла не поменялся!
Добавил новый класс - старый код работает без изменений

Абстрактные классы

Хотим заставить наследников реализовать метод - используем ABC. 
from abc import ABC, abstractmethod

class Animal(ABC):
    @abstractmethod
    def speak(self):
        pass                  # реализации нет!

class Dog(Animal):
    def speak(self):
        print("Гав!")

Animal()   # TypeError - нельзя создать абстрактный
Dog()      # OK

Декораторы в ООП

@property, @classmethod, @staticmethod

Что такое декоратор

Декоратор - это надстройка над методом. Пишется через @ над def. 
class Dog:
    @staticmethod       # <-- декоратор
    def bark():
        print("Гав!")
Декоратор меняет поведение метода. Разберём три главных для ООП

@staticmethod - без self

class MathUtils:
    @staticmethod
    def is_even(n):
        return n % 2 == 0

MathUtils.is_even(4)   # True
MathUtils.is_even(7)   # False
Обычная функция, живущая внутри класса для организации
Не нужен ни объект, ни self

Когда нужен @staticmethod

class Temperature:
    def __init__(self, celsius):
        self.celsius = celsius

    @staticmethod
    def c_to_f(c):
        return c * 9/5 + 32

# Утилита, не зависящая от конкретного объекта
print(Temperature.c_to_f(25))   # 77.0
Когда метод связан с классом по смыслу, но не работает с объектом

@classmethod - cls вместо self

class Dog:
    count = 0

    def __init__(self, name):
        self.name = name
        Dog.count += 1

    @classmethod
    def total(cls):        # cls - сам класс
        return cls.count

Dog("Рекс"); Dog("Шарик"); Dog("Бим")
print(Dog.total())   # 3
cls - это сам класс, не объект

Альтернативный конструктор

class Date:
    def __init__(self, day, month, year):
        self.day = day
        self.month = month
        self.year = year

    @classmethod
    def from_string(cls, text):
        d, m, y = text.split("-")
        return cls(int(d), int(m), int(y))

d = Date.from_string("21-04-2026")
print(d.year)   # 2026
Классический паттерн: from_string, from_dict, from_file

Три типа методов

Тип Первый параметр Видит
обычный self объект и класс
@classmethod cls только класс
@staticmethod нет ничего

@property - метод как атрибут

class Rectangle:
    def __init__(self, w, h):
        self.w = w
        self.h = h

    @property
    def area(self):
        return self.w * self.h

r = Rectangle(4, 5)
print(r.area)     # 20  - БЕЗ скобок!
Было r.area(), стало r.area - читается как атрибут, но вычисляется

Зачем @property

Без property

r.get_area()
user.get_full_name()

С property

r.area
user.full_name
Код выглядит как работа с данными, а не с функциями

Setter - присваивание с проверкой

class Person:
    def __init__(self, age):
        self._age = age

    @property
    def age(self):
        return self._age

    @age.setter
    def age(self, value):
        if value < 0:
            raise ValueError("Возраст не может быть отрицательным")
        self._age = value

p = Person(20)
p.age = 25       # OK, прошло setter
p.age = -5       # ValueError

Пример: температура

class Temperature:
    def __init__(self, c):
        self.celsius = c

    @property
    def celsius(self):
        return self._c

    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("Ниже абсолютного нуля нельзя")
        self._c = value

t = Temperature(25)
t.celsius = -500   # ValueError
@property + setter = защита от неправильных значений

@dataclass - автогенерация

from dataclasses import dataclass

@dataclass
class Point:
    x: int
    y: int

p1 = Point(1, 2)
p2 = Point(1, 2)

print(p1)          # Point(x=1, y=2) - __repr__
print(p1 == p2)    # True - __eq__
@dataclass сам генерирует __init__, __eq__, __repr__
Удобно для простых "контейнеров с данными"

Шпаргалка: декораторы ООП

Декоратор Что делает
@staticmethod метод без self - просто функция в классе
@classmethod метод с cls (сам класс) - альтернативные конструкторы
@property метод читается как атрибут: obj.area
@x.setter валидация при obj.x = value
@dataclass автоматически __init__, __eq__, __repr__

SOLID

5 принципов хорошего дизайна

Зачем эти принципы

🔧
Код легко изменять
📖
Код легко читать
🧪
Код легко тестировать
♻️
Код легко переиспользовать
SOLID - 5 первых букв пяти правил

S - Single Responsibility

Принцип единственной ответственности

Один класс - одна ответственность.
Причина изменить класс должна быть только одна

SRP: плохо и хорошо

❌ Всё в одном

class User:
    def save_to_db(self):
        ...
    def send_email(self):
        ...
    def render_html(self):
        ...

✅ Разнесли

class User: ...
class UserRepo:      # БД
    def save(u): ...
class Mailer:        # почта
    def send(u): ...
Поменялась БД - правим только UserRepo

O - Open / Closed

Принцип открытости/закрытости

Открыт для расширения, закрыт для изменения.
Добавляем новое поведение наследником, не трогая старый код

OCP: оплата

❌ if/else на каждый тип

def pay(kind, amount):
    if kind == "card":
        ...
    elif kind == "cash":
        ...
    # новый тип - правим функцию

✅ Наследники

class Payment:
    def pay(self): ...

class Card(Payment): ...
class Cash(Payment): ...
class Crypto(Payment): ...
# новый тип = новый класс

L - Liskov Substitution

Принцип подстановки Барбары Лисков

Наследник должен полностью заменять родителя.
Если код работал с Animal - он должен работать с любым наследником без сюрпризов.

LSP: классический контрпример

class Rectangle:
    def set_width(self, w): self.w = w
    def set_height(self, h): self.h = h

class Square(Rectangle):   # квадрат = прямоугольник?
    def set_width(self, w):
        self.w = w
        self.h = w         # изменили ОБЕ стороны

r = Square()
r.set_width(5)
r.set_height(10)
# у Square ширина != 5, это сюрприз!
Наследование не всегда совпадает с математикой

I - Interface Segregation

Принцип разделения интерфейса

Много маленьких интерфейсов лучше одного большого. 
# ❌ толстый базовый класс
class Device:
    def print(): ...
    def scan(): ...
    def fax(): ...      # старый факс, никому не нужен

# ✅ разделили
class Printer:  def print(): ...
class Scanner: def scan(): ...
class Fax:     def fax(): ...
Принтер без сканера не должен реализовывать scan()

D - Dependency Inversion

Принцип инверсии зависимостей

Зависим от абстракций, не от конкретных классов. 
# ❌ жёсткая зависимость
class App:
    def __init__(self):
        self.log = FileLogger()   # только в файл

# ✅ передаём любой логгер
class App:
    def __init__(self, logger):
        self.log = logger

app = App(FileLogger())
app = App(ConsoleLogger())   # легко заменить

Памятка SOLID

S Single Responsibility один класс = одна задача
O Open / Closed расширять, не менять
L Liskov наследник заменяет родителя
I Interface Segregation маленькие интерфейсы
D Dependency Inversion зависеть от абстракций
Не учить наизусть. Главное - знать, что такие правила есть
