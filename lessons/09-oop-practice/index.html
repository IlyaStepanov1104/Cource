1. Класс, объект, self

class Dog:
    def __init__(self, name, age):
        self.name = name          # атрибут
        self.age = age

    def bark(self):               # метод
        print(f"{self.name}: Гав!")

rex = Dog("Рекс", 3)              # создали объект
rex.bark()                        # Рекс: Гав!
__init__ задаёт стартовые атрибуты, self - сам объект

2. Наследование + super()

class Animal:
    def __init__(self, name):
        self.name = name

class Dog(Animal):
    def __init__(self, name, breed):
        super().__init__(name)    # зовём родителя
        self.breed = breed

rex = Dog("Рекс", "Хаски")
print(rex.name, rex.breed)        # Рекс Хаски
Переопределил __init__ - обязательно зови super().__init__()

3. Полиморфизм + ABC

from abc import ABC, abstractmethod

class Animal(ABC):
    @abstractmethod
    def speak(self): pass

class Dog(Animal):
    def speak(self): print("Гав!")
class Cat(Animal):
    def speak(self): print("Мяу")

for a in [Dog(), Cat()]:
    a.speak()                     # один вызов, разный звук
ABC - заставит наследника реализовать speak

4. @property + setter

class Person:
    def __init__(self, age):
        self._age = age

    @property
    def age(self):
        return self._age

    @age.setter
    def age(self, value):
        if value < 0:
            raise ValueError("нельзя отрицательный")
        self._age = value

p = Person(20)
p.age = 25        # OK
p.age = -5        # ValueError
Атрибут с валидацией - это и есть @property + setter

5. @classmethod + @staticmethod

class Date:
    def __init__(self, d, m, y):
        self.d, self.m, self.y = d, m, y

    @classmethod
    def from_string(cls, s):          # альтернативный конструктор
        d, m, y = s.split("-")
        return cls(int(d), int(m), int(y))

    @staticmethod
    def is_leap(year):                # просто утилита
        return year % 4 == 0

Date.from_string("21-04-2026")
Date.is_leap(2024)                    # True

Шпаргалка декораторов

Декоратор Первый параметр Когда применять
обычный метод self работа с конкретным объектом
@classmethod cls альтернативные конструкторы
@staticmethod нет утилита в пространстве класса
@property self метод читается как атрибут
@x.setter self, value валидация при присваивании

SOLID

5 правил хорошего дизайна

Представь

Ты пишешь программу - 100 строк. Всё понятно.
Проходит месяц - уже 1000 строк. Надо поменять одну вещь.
Меняешь в одном месте - ломается в трёх других.
Знакомо? Это называется технический долг

Зачем SOLID

Код на ООП - это не просто «классы есть, значит хорошо».
1
Код легко менять - правка в одном месте, а не в десяти
2
Код легко тестировать - можно проверять по кусочкам
3
Код легко расширять - новая фича без ломки старого

Откуда взялся SOLID

Роберт Мартин (Uncle Bob) - в 2000-х сформулировал 5 принципов хорошего ООП-дизайна.
Позже Майкл Фезерс придумал акроним SOLID - по первым буквам этих принципов.
Это не магия, а собранный опыт того, какой код дольше живёт

5 букв

S Single Responsibility один класс - одна задача
O Open / Closed расширяй, не меняй
L Liskov наследник заменяет родителя
I Interface Segregation много мелких интерфейсов
D Dependency Inversion зависим от абстракций

S - Single Responsibility

Принцип единственной ответственности

Один класс должен иметь только одну причину для изменения.
Если в описании класса есть слово «и» - скорее всего, он делает слишком много
Пример: класс User который умеет хранить данные и сохранять в БД и отправлять почту

SRP: плохо

class Report:
    def __init__(self, data):
        self.data = data

    def calculate_total(self):         # расчёт
        return sum(self.data)

    def format_html(self):             # отображение
        return f"<h1>Итог: {self.calculate_total()}</h1>"

    def save_to_file(self, path):      # работа с файлами
        with open(path, "w") as f:
            f.write(self.format_html())

    def send_email(self, to):          # работа с почтой
        smtp.send(to, self.format_html())
Четыре причины менять этот класс: расчёт, HTML, файлы, почта

SRP: хорошо

class Report:                    # только данные
    def __init__(self, data):
        self.total = sum(data)

class ReportFormatter:           # только форматирование
    def to_html(self, report):
        return f"<h1>Итог: {report.total}</h1>"

class FileSaver:                 # только файлы
    def save(self, content, path):
        with open(path, "w") as f:
            f.write(content)

class EmailSender:               # только почта
    def send(self, to, content):
        smtp.send(to, content)
Поменялся формат HTML? Правим только ReportFormatter

SRP: как понять, что нарушил

  • В описании класса встречается «и» - «считает и сохраняет»
  • Класс больше ~200 строк
  • В одном файле перемешаны разные темы: бизнес-логика и работа с БД
  • Меняешь одно - ломается другое
  • Тестировать сложно: чтобы проверить расчёт, нужна настоящая БД

O - Open / Closed

Принцип открытости / закрытости

Открыт для расширения, закрыт для изменения.
Добавляй новое - пиши новый класс. Не трогай старые.
Если каждая новая фича требует переписать старую функцию - ты нарушаешь OCP

OCP: плохо

def pay(kind, amount):
    if kind == "card":
        charge_card(amount)
    elif kind == "cash":
        give_receipt(amount)
    elif kind == "crypto":
        send_btc(amount)
    # Добавили Apple Pay - снова правим функцию
    # Добавили СБП - снова правим функцию
    # Добавили PayPal - снова правим функцию
Каждый новый способ оплаты = правка той же функции. Риск сломать старое

OCP: хорошо

class Payment(ABC):
    @abstractmethod
    def pay(self, amount): pass

class CardPayment(Payment):
    def pay(self, amount): charge_card(amount)

class CashPayment(Payment):
    def pay(self, amount): give_receipt(amount)

class CryptoPayment(Payment):
    def pay(self, amount): send_btc(amount)

# Новый тип - новый класс. Старые не трогаем
class ApplePayPayment(Payment):
    def pay(self, amount): apple_charge(amount)
Код, использующий Payment, не меняется - работает с любым наследником

OCP: как достигается

  • Наследование + полиморфизм - новый тип = новый класс
  • Абстрактные классы - контракт для наследников
  • Функции как аргументы - передаём поведение снаружи
Если ловишь себя на цепочке if type == ... - это сигнал подумать об OCP

L - Liskov Substitution

Принцип подстановки Барбары Лисков

Наследник должен вести себя как родитель - без сюрпризов.
Код, написанный для Animal, должен работать с любым его наследником.
Автор - Барбара Лисков, лауреат премии Тьюринга, 2008

LSP: пингвин не летает

class Bird:
    def fly(self):
        print("лечу")

class Penguin(Bird):
    def fly(self):
        raise NotImplementedError("пингвины не летают!")

def travel(birds):
    for b in birds:
        b.fly()                    # ожидаем, что все полетят

travel([Eagle(), Sparrow()])       # OK
travel([Eagle(), Penguin()])       # БАХ
Penguin наследует Bird, но ломает контракт: fly() не работает

LSP: как исправить пингвина

class Bird:                       # общие свойства всех птиц
    def eat(self): ...

class FlyingBird(Bird):           # отдельный класс для летающих
    def fly(self): ...

class Eagle(FlyingBird): ...
class Sparrow(FlyingBird): ...
class Penguin(Bird): ...          # не наследует fly

def travel(birds: list[FlyingBird]):
    for b in birds:
        b.fly()                   # гарантия, что полетит
Разделили иерархию - пингвин больше не обещает то, чего не умеет

LSP: признаки нарушения

  • Наследник кидает NotImplementedError в методе родителя
  • В коде проверяется isinstance(obj, Penguin) перед вызовом
  • У наследника более строгие требования к аргументам
  • Наследник возвращает принципиально другой тип
Наследование по принципу «мне тут удобно взять готовое» - частая причина нарушения LSP

I - Interface Segregation

Принцип разделения интерфейса

Много маленьких интерфейсов лучше одного большого.
Не заставляй класс реализовывать то, что ему не нужно.

ISP: плохо

class Device(ABC):
    @abstractmethod
    def print(self, doc): pass
    @abstractmethod
    def scan(self): pass
    @abstractmethod
    def fax(self): pass

class SimplePrinter(Device):
    def print(self, doc): print_real(doc)
    def scan(self):
        raise NotImplementedError   # нет сканера
    def fax(self):
        raise NotImplementedError   # факса тоже нет
Принтер обязан реализовать scan и fax, хотя не умеет

ISP: хорошо

class Printer(ABC):
    @abstractmethod
    def print(self, doc): pass

class Scanner(ABC):
    @abstractmethod
    def scan(self): pass

class Fax(ABC):
    @abstractmethod
    def fax(self): pass

class SimplePrinter(Printer):        # только печать
    def print(self, doc): ...

class MFU(Printer, Scanner, Fax):    # принтер+сканер+факс
    def print(self, doc): ...
    def scan(self): ...
    def fax(self): ...
Каждый класс реализует только то, что умеет

ISP в Python

В Python нет «интерфейсов» как в Java. Что использовать:
  • Маленькие ABC с 1-2 методами
  • Duck typing - важен не тип, а наличие метода
  • Protocol из typing - «всё, что имеет .print()»
Главная идея: не навязывать методы, которые классу не нужны

D - Dependency Inversion

Принцип инверсии зависимостей

Зависим от абстракций, а не от конкретных классов.
Аналогия: розетка. Её стандарт не зависит от того, какая именно лампа будет подключена.

DIP: плохо

class FileLogger:
    def log(self, msg):
        with open("app.log", "a") as f:
            f.write(msg + "\n")

class App:
    def __init__(self):
        self.logger = FileLogger()       # жёстко прибили

    def run(self):
        self.logger.log("старт")

# Захотели в консоль? Меняем App
# Захотели отключить логи в тестах? Меняем App
App намертво связан с FileLogger

DIP: хорошо

class Logger(ABC):                   # абстракция
    @abstractmethod
    def log(self, msg): pass

class FileLogger(Logger): ...
class ConsoleLogger(Logger): ...
class SilentLogger(Logger):          # для тестов
    def log(self, msg): pass

class App:
    def __init__(self, logger: Logger):
        self.logger = logger          # получаем снаружи

    def run(self):
        self.logger.log("старт")

app = App(FileLogger())
test_app = App(SilentLogger())        # легко подменили

DIP: что даёт

  • Тестируемость - подставляем fake-объект вместо реальной БД
  • Гибкость - меняем реализацию без переписывания App
  • Независимые слои - бизнес-логика не знает про SQL или HTTP
Передавай зависимости через конструктор, а не создавай внутри

Памятка SOLID

Буква Правило Ключевая мысль
S Single Responsibility одна причина для изменения
O Open / Closed расширяй новым классом
L Liskov наследник без сюрпризов
I Interface Segregation маленькие интерфейсы
D Dependency Inversion зависим от абстракций

Важное про SOLID

  • Это ориентир, а не догма
  • В скрипте на 50 строк SOLID не нужен - будет только хуже
  • SOLID включаешь, когда проект растёт и менять становится больно
  • Нарушил? Не страшно. Главное - понимать, зачем
SOLID - это про то, как сделать код дружелюбным к изменениям
